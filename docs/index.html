Triple Inverted Pendulum

A simple walkthrough of how the physics simulation works, written as plain Python pseudocode.

Background — What & why

●

What is a triple inverted pendulum, and why is it hard?

A regular pendulum hangs downward — give it a nudge and gravity passively pulls it back to centre. Flip it upside-down and gravity does the opposite: it amplifies any small lean, making it grow exponentially. This is called an actively unstable equilibrium. Now stack three such links end-to-end above a motorised cart and you have one of the most challenging benchmark problems in control engineering.

4 degrees of freedom 3 coupled unstable modes single control input (cart force F) exponential divergence without control

A 3° initial lean at the top link will cause the tip to pass 90° in roughly 0.3 seconds — faster than human reaction time. The only practical stabilisation strategy is a computer-calculated optimal gain matrix (LQR), applied at 100 Hz.

Real hardware: This simulation models an actual GoBilda lead-screw actuator (1120 RPM motor, 8 mm lead, 85% mechanical efficiency). The motor's torque-speed curve and back-EMF are included, so the physics directly reflect what the physical build can and cannot do.

The Lagrangian method is used to derive the equations of motion. Instead of drawing free-body diagrams and tracking every contact force at every pivot, we write expressions for the total kinetic and potential energy of the system and apply one master formula — the Euler-Lagrange equation — which automatically produces all four equations of motion.

Step 1 — Define the system

1

Describe the physical setup

We describe the whole system with just four numbers called generalised coordinates. Instead of tracking the X and Y position of every single point, we only need:

  • x — where the cart is on the rail (metres, positive = right)
  • θ₁ — how far link 1 is tilted from straight-up (radians)
  • θ₂ — tilt of link 2
  • θ₃ — tilt of link 3

Everything else (the XY position of every joint and mass) can be calculated from these four numbers.

## ── STEP 1: Define the system parameters ───────────────────────────────

# All the physical constants you can change to test different setups
M_cart = 2.0   # kg — how heavy is the cart?
l1, m1 = 0.30, 0.15   # link 1: length (m), mass (kg)
l2, m2 = 0.25, 0.10   # link 2
l3, m3 = 0.20, 0.08   # link 3
g = 9.81               # gravity (m/s²)

# The state of the whole system is 8 numbers:
#   positions: x, theta1, theta2, theta3
#   velocities: x_dot, theta1_dot, theta2_dot, theta3_dot
state = [x, theta1, theta2, theta3,
         x_dot, theta1_dot, theta2_dot, theta3_dot]

Step 2 — Physics: equations of motion

2

Calculate accelerations from forces (Lagrangian mechanics)

The Lagrangian method gives us one master equation that automatically produces all four equations of motion. It works from energy instead of drawing free-body diagrams:

T = kinetic energy (motion) V = potential energy (height) L = T − V

After doing the calculus (see README for full derivation), the result is a compact matrix equation:
M(q) · q̈ = F_ext − Coriolis(q, q̇) − Gravity(q) − Damping

  • M(q) — 4×4 mass matrix (how inertia is spread across all joints)
  • Coriolis — centrifugal / coupling forces when links spin fast
  • Gravity — torque gravity applies to each tilted link
  • F_ext — cart force from the motor (our control handle)

We solve for the accelerations by rearranging:

## ── STEP 2: Compute accelerations at every timestep ────────────────────

def equations_of_motion(state, F_cart):
    x, th1, th2, th3, x_dot, th1_dot, th2_dot, th3_dot = state

    # ── 2a. Build the 4×4 mass matrix M ──────────────────────────────────
    # (how hard it is to accelerate each coordinate)
    # Diagonal = self-inertia; off-diagonal = coupling between links
    M = build_mass_matrix(th1, th2, th3)   # depends on current angles

    # ── 2b. Coriolis / centrifugal vector h ───────────────────────────────
    # These appear because the mass matrix changes as the links rotate.
    # Think: a spinning link "flings" the one above it sideways.
    # They are proportional to the SQUARE of angular velocities (θ_dot²),
    # so they are small near the upright and large during fast swings.
    h = coriolis_vector(th1, th2, th3, th1_dot, th2_dot, th3_dot)

    # ── 2c. Gravity torque vector G ───────────────────────────────────────
    # Gravity pulls each leaning link further away from vertical.
    # G[i] = -alpha_i * g * sin(theta_i)
    # When theta > 0 (leaning right), G < 0 → pulls it further right.
    # This is the INSTABILITY — gravity is a destabilising force here.
    G = gravity_vector(th1, th2, th3)

    # ── 2d. Damping (friction) ────────────────────────────────────────────
    D = [cart_friction * x_dot,
         joint_damping * th1_dot,
         joint_damping * th2_dot,
         joint_damping * th3_dot]

    # ── 2e. Solve  M · q_ddot = F_ext - h - G - D  ───────────────────────
    F_ext = [F_cart, 0, 0, 0]   # motor force acts only on the cart
    rhs = F_ext - h - G - D
    q_ddot = solve(M, rhs)      # numpy.linalg.solve — fast, no explicit M⁻¹

    # Returns [x_ddot, th1_ddot, th2_ddot, th3_ddot,
    #          x_dot,  th1_dot,  th2_dot,  th3_dot ]
    return q_ddot + [x_dot, th1_dot, th2_dot, th3_dot]

Step 3 — Motor model

3

Convert motor command → cart force

The GoBilda Yellow Jacket brushed DC motor follows a simple linear torque-speed curve. The faster the cart already moves, the less force the motor can deliver (back-EMF effect). A lead screw then converts shaft rotation into linear cart force.

  • Primary motor: 1120 RPM → max 0.149 m/s (8 mm lead)
  • Secondary motor: 312 RPM → max 0.042 m/s, but 2× the torque
## ── STEP 3: Motor — shaft torque → cart force ───────────────────────────

FREE_SPEED_RPM  = 1120          # no-load speed at full voltage
STALL_TORQUE_Nm = 2.8           # maximum torque (at zero speed)
LEAD_m          = 0.008         # 8 mm lead screw (metres per revolution)
EFFICIENCY      = 0.85          # thread friction loss

def motor_force(command, cart_velocity):
    """
    command : float in [-1, +1]  (PWM duty cycle from controller)
    cart_velocity : current cart speed in m/s

    Returns: force applied to cart in Newtons
    """
    # Convert cart speed back to shaft speed
    omega_free   = FREE_SPEED_RPM * 2*pi / 60   # rad/s
    omega_shaft  = cart_velocity / (LEAD_m / (2*pi))  # rad/s

    # Torque-speed curve:  tau = tau_stall * (command - omega_shaft/omega_free)
    # At stall (omega_shaft=0): tau = tau_stall * command  (max torque)
    # At free-run (omega_shaft=omega_free): tau = 0         (no torque)
    tau = STALL_TORQUE_Nm * (command - omega_shaft / omega_free)

    # Lead screw: F = tau * (2*pi / lead) * efficiency
    force = tau * (2*pi / LEAD_m) * EFFICIENCY
    return force

Step 4 — Controller (LQR)

4

Compute the control force (LQR — optimal automatic gains)

A plain PD controller is very hard to tune for a triple pendulum — the three links couple together in complex ways. Instead we use LQR (Linear Quadratic Regulator), which automatically calculates the best possible gains by solving a single equation.

LQR linearises the physics at the upright position (where sin θ ≈ θ), then finds the gain matrix K that minimises a cost function balancing how much the links lean vs. how much force we use. 

## ── STEP 4: LQR controller ──────────────────────────────────────────────

# ── 4a. Linearise the physics at the upright equilibrium ─────────────────
# At theta=0: sin(theta)≈theta, cos(theta)≈1, Coriolis≈0
# The result is a simple linear system:   z_dot = A*z + B*u
A, B = linearise_at_upright(params)   # 8×8 and 8×1 matrices

# ── 4b. Choose cost weights ───────────────────────────────────────────────
# Q = how much do we care about each state being non-zero?
# R = how much do we care about using large forces?
#       State:   [ x,   th1,   th2,   th3,  x_dot, th1_dot, th2_dot, th3_dot ]
Q = diagonal(    [10,  2000,  2000,  2000,     1,     100,     100,     100  ])
R = 0.001   # small = controller is allowed to use large forces

# ── 4c. Solve the Riccati equation for the gain matrix K ─────────────────
# scipy finds the matrix P such that:
#   A^T P + P A - P B R^-1 B^T P + Q = 0
# Then: K = R^-1 * B^T * P   (shape: 1 × 8)
P = scipy.linalg.solve_continuous_are(A, B, Q, R)
K = (1/R) * B.T @ P

# ── 4d. At every timestep, apply the optimal control law ─────────────────
def lqr_force(state, x_ref=0.0):
    z = state.copy()
    z[0] -= x_ref          # express cart position relative to target
    force = -(K @ z)       # single matrix-vector multiply → one number
    return clamp(force, -MAX_FORCE, +MAX_FORCE)

# All 8 gains are found automatically.
# The signs and magnitudes are globally optimal for the chosen Q and R.
# No manual trial-and-error needed.

Step 5 — Run the simulation

5

Step the simulation forward in time

We use a Zero-Order-Hold (ZOH) loop: at every control step the controller picks a force, we hold it constant, and then let the physics integrator advance time with that fixed force. This is exactly what real microcontrollers do (sample → compute → output → wait → repeat). 

## ── STEP 5: Main simulation loop ────────────────────────────────────────

# Starting conditions — pendulum nearly upright, tiny lean on each link
state = [0.0,           # x:  cart at centre
         radians(3),    # th1: 3° lean
         radians(2),    # th2: 2° lean
         radians(1.5),  # th3: 1.5° lean
         0, 0, 0, 0]    # all velocities zero

dt      = 0.01   # control step = 10 ms  (100 Hz)
t_end   = 5.0    # simulate 5 seconds
history = []     # store results for plotting

t = 0.0
while t < t_end:

    # ── 5a. Controller decides the force ─────────────────────────────────
    F_cart = lqr_force(state, x_ref=0.0)   # wants cart at x=0

    # ── 5b. Convert to motor command, then apply back-EMF model ──────────
    command = F_cart / MAX_FORCE             # normalise to [-1, +1]
    F_actual = motor_force(command, state[4])  # real force (back-EMF limited)

    # ── 5c. Integrate physics for this control step ───────────────────────
    # scipy RK45 adaptive solver — automatically picks sub-steps for accuracy
    def ode(t, s):
        return equations_of_motion(s, F_actual)   # F_actual is constant here

    solution = scipy.integrate.solve_ivp(ode, [t, t+dt], state,
                                         method='RK45', rtol=1e-6)
    state = solution.y[:, -1]   # state at end of this step

    # ── 5d. Record results ────────────────────────────────────────────────
    history.append({
        't':     t,
        'x':     state[0],
        'th1':   degrees(state[1]),
        'th2':   degrees(state[2]),
        'th3':   degrees(state[3]),
        'force': F_actual,
    })

    t += dt   # advance clock

# history now contains the full time series — plot or animate it

Deep dive — The mathematics

∑

Lagrangian mechanics — energy instead of forces

The Lagrangian L = T − V (kinetic minus potential energy) encodes everything the equations of motion need. Applying the Euler-Lagrange formula to each coordinate automatically produces all four equations of motion — no contact forces at pivots, no constraint equations.

Kinetic energy T — each body contributes translational and rotational terms:

T = ½ M_cart·ẋ² + ½ m₁·(ẋ_cm1² + ẏ_cm1²) + ½ I₁·θ̇₁² + ½ m₂·(ẋ_cm2² + ẏ_cm2²) + ½ I₂·θ̇₂² + ½ m₃·(ẋ_cm3² + ẏ_cm3²) + ½ I₃·θ̇₃² where Iᵢ = mᵢ·lᵢ²/3 (uniform rod, rotating about one end)

Each centre-of-mass velocity (ẋ_cmi, ẏ_cmi) depends on the cart motion ẋ plus the rotation of every link below link i. This coupling is why the mass matrix has off-diagonal entries — pushing the cart simultaneously accelerates all the links.

Potential energy V — only gravity, measured upward from the rail:

V = m₁·g·(l₁/2)·cos θ₁ + m₂·g·(l₁·cos θ₁ + (l₂/2)·cos θ₂) + m₃·g·(l₁·cos θ₁ + l₂·cos θ₂ + (l₃/2)·cos θ₃)

At θ = 0 (perfectly upright), V is at a maximum. Any small lean lowers V, releasing energy into angular velocity — this is the mathematical signature of instability. Gravity is not a restoring force here; it is a destabilising one.

M

The 4×4 mass matrix — encoding inertia and coupling

After expanding T and collecting terms grouped by pairs of generalised velocities, the equations of motion take the compact form M(q)·q̈ = Q − h − G − D. The symmetric 4×4 matrix M(q) encodes how every force and torque is distributed across all four coordinates.

Six constant coupling coefficients (pre-computed once from masses and lengths):

SymbolFormulaPhysical meaning
α₁m₁·(l₁/2) + (m₂+m₃)·l₁Effective mass coupling cart translation to link 1 rotation
α₂m₂·(l₂/2) + m₃·l₂Effective mass coupling cart translation to link 2 rotation
α₃m₃·(l₃/2)Effective mass coupling cart translation to link 3 rotation
β₁₂(m₂·(l₂/2) + m₃·l₂)·l₁Coupling inertia between links 1 and 2
β₁₃m₃·(l₃/2)·l₁Coupling inertia between links 1 and 3
β₂₃m₃·(l₃/2)·l₂Coupling inertia between links 2 and 3

Key entries of M (upper triangle only; matrix is symmetric):

M[0,0] = M_cart + m₁ + m₂ + m₃ — total system mass (cart + all links) M[0,1] = α₁·cos θ₁ — cart–link1 coupling M[0,2] = α₂·cos θ₂ — cart–link2 coupling M[0,3] = α₃·cos θ₃ — cart–link3 coupling M[1,1] = m₁·(l₁/2)² + I₁ + (m₂+m₃)·l₁² — link1 self-inertia M[1,2] = β₁₂·cos(θ₁−θ₂) — link1–link2 coupling M[1,3] = β₁₃·cos(θ₁−θ₃) — link1–link3 coupling M[2,2] = m₂·(l₂/2)² + I₂ + m₃·l₂² — link2 self-inertia M[2,3] = β₂₃·cos(θ₂−θ₃) — link2–link3 coupling M[3,3] = m₃·(l₃/2)² + I₃ — link3 self-inertia

The cosine factors in the off-diagonal entries capture how the angle between adjacent links determines how strongly they pull on each other. At large angles the coupling decreases — but by then the linearised LQR controller is no longer valid.

h G

Coriolis forces h(q,q̇) and gravity vector G(q)

Coriolis / centrifugal vector h — arises when rotating links "fling" each other sideways. Proportional to angular velocity squared, so it is negligible near the upright (small θ̇) but significant during fast swings:

h[0] = −α₁·sin θ₁·θ̇₁² − α₂·sin θ₂·θ̇₂² − α₃·sin θ₃·θ̇₃² h[1] = +β₁₂·sin(θ₁−θ₂)·θ̇₂² + β₁₃·sin(θ₁−θ₃)·θ̇₃² h[2] = −β₁₂·sin(θ₁−θ₂)·θ̇₁² + β₂₃·sin(θ₂−θ₃)·θ̇₃² h[3] = −β₁₃·sin(θ₁−θ₃)·θ̇₁² − β₂₃·sin(θ₂−θ₃)·θ̇₂²
h[0] is the centripetal reaction on the cart: when any link spins, its pivot sweeps outward, pushing the cart sideways. This is why the LQR must continuously move the cart even when the angles are nearly zero.

Gravity vector G = ∂V/∂q — the torque gravity applies to each leaning link:

G[0] = 0 — gravity is vertical; x position does not affect height G[1] = −α₁·g·sin θ₁ — gravity tips link 1 further over G[2] = −α₂·g·sin θ₂ — gravity tips link 2 further over G[3] = −α₃·g·sin θ₃ — gravity tips link 3 further over

The negative signs mean: when θ > 0 (link leans right), G < 0. In the equation of motion M·q̈ = Q − h − G − D, subtracting a negative G adds a positive angular acceleration — gravity makes things worse. This is the mathematical definition of the instability.

K

LQR: the optimal control law — from Riccati equation to gain matrix

LQR finds the gain matrix K that minimises a cost integral over all future time:

J = ∫₀^∞ ( zᵀ·Q·z + u·R·u ) dt z = [x, θ₁, θ₂, θ₃, ẋ, θ̇₁, θ̇₂, θ̇₃] (8-element state vector) u = F_cart (single scalar control input) Q = 8×8 diagonal matrix (state deviation penalties — you choose these) R = scalar (control effort penalty — you choose this)

Larger Q weights → controller reacts more aggressively to state deviations. Larger R → gentler controller that uses less force (may fail to stabilise at large angles).

The physics is first linearised at the upright equilibrium (all angles and velocities zero): sin θ ≈ θ, cos θ ≈ 1, Coriolis ≈ 0. This yields a linear state-space model:

ż = A·z + B·u (A is 8×8, B is 8×1 — both computed analytically)

Then the algebraic Riccati equation is solved once (offline, before the simulation starts):

Aᵀ·P + P·A − P·B·R⁻¹·Bᵀ·P + Q = 0 → solution P (8×8) Optimal gain matrix: K = R⁻¹ · Bᵀ · P (shape 1×8)

At every 10 ms control step, the force is computed in a single matrix multiply:

F_cart = − K · z (1×8 matrix times 8×1 vector = one scalar)
Why LQR beats hand-tuned PD: A triple pendulum has four coupled unstable modes. Tuning P and D gains independently for each link ignores cross-coupling and easily violates global stability conditions. LQR finds the globally optimal gains for the chosen Q and R in one calculation — and always produces a provably stable controller as long as the system is controllable. Stability is verified by checking that all eigenvalues of the closed-loop matrix A−B·K have negative real parts.

↓

↓

The Zero-Order Hold (ZOH) integration strategy is critical for correctness with an adaptive ODE solver. At the start of each 10 ms control interval the controller evaluates the state and outputs a constant force. The ODE solver (RK45, adaptive step size) then integrates the physics with that fixed force — making the ODE function a pure, stateless function of (t, state). This is exactly how a real microcontroller operates: sample → compute → output → wait → repeat.

Key lesson: AI is excellent at mechanics — translating mathematical formulas into code, generating boilerplate, drafting explanations. It is less reliable for engineering judgment: choosing cost weights, deciding whether motor bandwidth is sufficient, interpreting eigenvalues, and catching subtle sign errors in physics derivations. All of those required human review and correction.

The workflow was iterative: Copilot generated a first draft of each class, human review identified correctness issues (four revision cycles on the physics alone), and Copilot revised. The free_swing energy conservation test — verifying that total mechanical energy is approximately constant with no control applied — was the key correctness benchmark throughout development.

This project is a good example of AI-assisted engineering: the AI dramatically accelerated the implementation, especially the tedious Lagrangian algebra and repetitive boilerplate, while a human provided the domain knowledge, hardware context, and quality control.
